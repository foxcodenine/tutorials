3.1 Introduction

While it's possible to use Linux without a Graphical User Interface (GUI), there are applications that benefit from being able to work visually, such as graphic or video composition or editing. Many Linux servers, however, do not use the GUI because the applications for which they are used do not benefit from a GUI display and add additional overhead. GUIs are also frowned upon in a virtual machine environment; many more instances of a virtual machine can be loaded when a GUI is not added to the system.

GUIs are intended to provide a user with a visual way of issuing commands to the computer through the use of Windows, Icons, Menus, and Pointers (WIMP). As opposed to the Command Line Interface (CLI) or shell, which is the way commands have traditionally been issued to the operating system.

When a Linux distribution is installed, it will normally have a means of installing and configuring one of the many possible GUI environments. For Microsoft Windows and Mac OSX, the GUIs are proprietary creations, and users are discouraged from altering functionality. With Linux, the GUI components are open source software products and as such, there is a great deal of customization that is permitted. The primary GUI server used by Linux has historically been based upon the X Window project, or simply X. (It is also frequently referred to informally as the X Window system or X Windows). The interface provided to the user is called a desktop. This is software that works with the X Window system to provide an easy way for the user to interact with the operating system.

Even though the tools for installing and configuring the X Window system have made great strides over time, there are still times when an administrator may need to know how to configure the X Window system manually. In this unit, you will learn about X Window components, and how to install and configure the X Window system using tools as well as manually.

3.2 Components

The following are some of the more common components of the X Window system:

  1. Hardware drivers
  2. X server
  3. X client
  4. Window managers
  5. Display manager
  6. Widget/toolkit libraries

3.2.1 Hardware Drivers

The hardware drivers include packages that define graphics adapters and input devices. Common graphics adapter (also referred to as video card, graphics card, or display card) libraries would include those from vendors such as AMD, Intel, Nvidia, and others. Common input devices, for example, a mouse and keyboard could be vendor-specific such as Apple, Logitech, Synaptic, or VIA. Input types can also be vendor-neutral, such as Bluetooth.

3.2.2 X Server

The X Window system uses a server-client model to manage the GUI on a local machine or a network of computers. The design of X Window is network-oriented with the X11 protocol allowing the user to have the X client application, like a terminal application, located either remotely or locally on the same system as the X server. This, in turn, allows the X client application to talk to and make requests for information from the X server, which returns the requested information to the X client. The main executable for the X server is a program simply known as X.

3.2.3 X Client

Although the concept of an application being a client to a local X server might seem confusing at first, it makes sense when you think that the server is the component that updates the display and handles keyboard and mouse input from the user. The X client could also be running on a remote machine where the screen is being updated with output from an X application running locally.

Window Manager

It is the job of the window manager to manage the placement of windows and determine which window components (menus, mouse pointers, etc.) will be visible. The window manager is also an X client with the special privilege of being able to manage other X clients. To think of X Window as a complete GUI would be a big mistake. Instead, X Window defines the X11 protocol for communication and graphics primitives. This protocol defines only the basic rules by which GUI-based devices communicate. By design, X Window does not specify what windows, buttons, icons, menus, pointers, login screens, etc., look like.

As window managers have become more sophisticated, some of them may require a video card and display monitor that support accelerated graphics routines. Obtaining accelerated graphics data may present one of the more difficult challenges in configuring X, as will be discussed later in this unit.

3.2.4 Display Manager

The display manager component is responsible for allowing the user to perform a graphical login making use of a network login protocol, the X Display Manager Control Protocol (XDMCP), and starting the window manager. Often, a list of all available users may be presented. The user can click on the icon representing their account and type their password to move from the greeter to the desktop environment created by the window manager. Typical display managers currently in use include the X Display Manager (XDM), the KDE Display Manager (KDM), the GNOME Display Manager (GDM), and the MDM Display Manager (MDM; a recursive acronym).⁠⁠ 

Typically, these display managers are included as part of the desktop environment:

  1. The GNOME desktop environment provides the gdm display manager.
  2. The KDE desktop environment provides the kdm display manager.
  3. If no desktop environment is installed, the xdm display manager is used.
  4. Some distributions have developed their own display manager. For example, the display manager for Ubuntu is called lightdm.

Each display manager mentioned above also has a set of configuration files:

  1. The gdm configuration files are found in the /etc/gdm directory. For some special settings, the /etc/gdm/custom.conf file is used.
  2. The kdm configuration files are found in the /etc/kdm directory.
  3. The xdm configuration files are found in the /etc/X11/xdm directory.

3.2.4.1 Turn the Display Manager On or Off

Whether or not a display manager will be on or off in most Linux distributions depends upon the default runlevel and the distribution. For Debian-derived distributions, this means that if the default runlevel is anything between 2 and 5, then a display manager will be started automatically if it is installed. For Debian, the /etc/inittab file defines the default runlevel within the second field of the following line:

id:2:initdefault:

In Red Hat-derived systems, the same line in the /etc/inittab file defines the default runlevel, and in runlevel 3, the display manager and graphical environment are not started, but if the default runlevel is 5, then the display manager will be started if all goes well:

id:5:initdefault:

3.2.4.2 Switch Display Managers

In order to switch from one display manager to another, there are a couple of techniques, depending on which distribution you use. For Debian-derived distributions, use the dpkg-reconfigure command for a package that provides a display manager. This allows the choice of which display manager to use by default. For example, executing the sudo dpkg-reconfigure kdm results in the following dialog box:

If the system is a Red Hat-derived Linux distribution, then the /etc/sysconfig/desktop file is used to set the default display manager and desktop environment. To use the gdm display manager and the GNOME desktop environment, this file should be set to contain the following entries:

DESKTOP="GNOME"
DISPLAYMANAGER="GNOME"

To use the kdm display manager and the KDE desktop environment, this file should be set to contain the following entries:

DESKTOP="KDE"
DISPLAYMANAGER="KDE"

It is not required that the desktop environment match the display manager that is being used. It would be possible to use the kdm display manager with the GNOME desktop or the gdm display manager with the KDE desktop, although since results may vary it’s best to match up the Desktop and Display Manager. Also, when multiple desktop environments are available, the user can choose which desktop to use by selecting that session with the display manager during the login process.

3.2.4.3 Change the Display Manager Greeting

The greeting or banner that is displayed for each display manager can be changed. For the gdm display manager, the settings for the banner are controlled by modifying an XML file. Rather than editing this file directly, the gconftool-2 is recommended for making changes. In Red Hat-derived distributions, the following series of commands would be performed by an administrator to set the banner to LPIC students only:

sysadmin@localhost:~$ init 3
sysadmin@localhost:~$ su -s /bin/sh gdm
sysadmin@localhost:~$ gconftool-2 --direct --config-source=xml:readwrite:$HOME/.gconf --type bool --set /apps/gdm/simple-greeter/banner_message_enable true
sysadmin@localhost:~$ gconftool-2 --direct --config-source=xml:readwrite:$HOME/.gconf\ --type string --set /apps/gdm/simple-greeter/banner_message_text "LPIC students only"
sysadmin@localhost:~$ exit
sysadmin@localhost:~$ init 5

For the kdm display manager, a banner or greeting can be set if the theme is disabled. An administrator can make changes to the /etc/kde/kdm/kdmrc file by modifying the UseTheme and GreetString keys to the values shown in the following:

UseTheme=false
GreetString=Message to display in banner

To make these changes effective, the administrator will need to restart the X server or restart the system.

3.2.4.4 Change Default Color Depth

The color depth for the display manager determines the maximum number of colors that can be displayed on the screen. Within the Screen section of the /etc/X11/xorg.conf file, the DefaultDepth should be set to a value that can be supported by the video card and monitor. The number that is specified is actually the exponent of the number 2, so a default depth of 16 indicates that 216 number of colors (or 65,536 colors) can be displayed:

Section       "Screen"
⁠⁠ 
Identifier    "Default Screen"
Monitor       "Monitor"
Device        "Video Card"
DefaultDepth  16
Modes         "1024x768"
EndSection

3.2.4.5 Widget/Toolkit Libraries

Typically, a window manager will use one library or widget toolkit for all of its widgets: icons, buttons, checkboxes, sliders, etc. Widget toolkits include the old and once proprietary Motif (which was standardized by the Common Desktop Environment used by UNIX), the modern, popular Qt (pronounced ‘cute’) toolkit used for C++ development, and the very common Gtk+ (GIMP toolkit) used for C development of graphical applications.

With the dazzling array of available components and graphical applications, it can be a daunting task to put together an efficient, stable, usable, and “visually appealing” environment. To aid in making this task easier, pre-tested combinations of components are available, which are known as Desktop Environments (note that some of these tools have abandoned the desktop metaphor completely). Two of the most popular desktop environments are:

  • GNOME “GuhNome”
  • KDE “KayDeeEee”

One of the benefits of installing the desktop manager with a desktop environment is that it will come bundled with a number of programs that use the same library for their widgets. As a result, all of the programs will show the same appearance and usability.

By only using one toolkit, the desktop environment makes efficient use of resources. Typical programs that may be bundled with the desktop environment include:

  • Window manager
  • Display manager
  • File manager
  • Text editor
  • Video player
  • Audio player
  • Image viewer
  • Web browser
  • Email client

The state of the Graphical User Interface (GUI) for Linux is always evolving. Thirty years after the introduction of the X Window project at the Massachusetts Institute of Technology (MIT), the future for the X11 protocol is uncertain.

Many of the features that were originally designed into the X server are no longer needed or used as frequently. The fonts that were originally loaded and displayed are often not used, as other fonts are now available to be rendered directly by the X client. Likewise, the ability to use graphics primitives such as lines and arcs has been surpassed with new libraries, which provide much more advanced capabilities.

Note

When studying for the LPIC-1 exam, keep in mind that the life-cycle for the exams is a three-year cycle, so newer technologies may not be included as testable items.

Hosted by freedesktop.org and sharing many of the same drivers as X, the Wayland protocol seems to be one of the more promising alternatives to X. Although the reference implementation of the Wayland protocol, known as Weston, could possibly replace the X server, the protocol currently allows for one or more X servers to be used for backward compatibility. With Weston, the display clients render their own graphics buffer, and the Weston server composites them onto the display. The Weston server replaces not only the X server but also the window management and compositing functions.

Note

As a result of these recent changes, preparing for the LPIC-1 exam objectives that relate to configuring aspects of the GUI is difficult, since distributions such as Debian, Ubuntu, or Fedora no longer use the GNOME desktop metaphor. Additionally, many of these distributions have incorporated major changes to components like GDM. At the time of this writing, Red Hat Enterprise Linux 6 and its derivatives like CentOS 7 and Scientific Linux 7 are the few remaining distributions that have not yet abandoned the GNOME desktop.

3.3 Configuring X Windows

There are many types of hardware that may be used in an X Windows configuration:

  1. Video cards
  2. Display monitors
  3. Keyboards
  4. Mice
  5. Pointing devices

The video card used is the most critical piece of hardware to be detected because if it doesn't work, then it won't be possible to see if anything else works. For laptop computers, if a video card doesn't work, the administrator won't be able to easily switch it for one that should. Therefore, when shopping for a laptop to use with Linux, be careful to verify the compatibility of the components.

For Debian-derived distributions, starting the system in any runlevel other than single-user mode will automatically attempt to start the X server. Recall that the X server is responsible for handling updates to the display and handles keyboard and mouse input from the user as well as communicating with X clients (applications). For Red Hat-derived systems, the X server is only started in runlevel 5 by default. Therefore, if an administrator does not want X Windows to start automatically, then the system can be started in single-user mode (or runlevel 1 which is the equivalent).

If there is any question about the hardware changes to the computer being compatible with X Windows, then running in a non-graphical mode is recommended. The X server and installed desktop environment can be started manually for testing when changes are being made.

To start in single-user mode, an administrator can press a key at boot time when the GRUB menu appears, select to Edit the kernel parameters, and add one of the following values: 1, s, S, or single. The resulting kernel line may appear something like the following with the runlevel specified as the final parameter:

kernel /boot/vmlinuz-3.1.25 root=/dev/sda2 ro 1

Booting the system with the runlevel specified as a kernel parameter causes the system to use that runlevel instead of the default runlevel. Starting in runlevel 1, or single-user mode, with the X server not started in Debian or Red Hat-derived distributions, makes it possible to troubleshoot the X server.

If the system is already running, you can attempt to access an alternative console by pressing Ctrl+Alt+F2. Provided the previous task was successful, it should be possible to log in to the system via the command line.

Note

Using F2 may not work on all systems. Try each F# key until you see a command line login.

After logging in to the system, switch to runlevel one by executing the following with root privileges:

root@localhost:~# telinit 1
root@localhost:~# startx

This approach is typically used to fully test the X environment because if the X server doesn't start, then the computer will be in a state to troubleshoot the issue.

Normally, most systems only configure the X server for one display; the identifier for this display will be referred to as 0. If additional displays are configured, they are numbered incrementally, so a second display would be referred to as 1. Later in this module, the display number will be referred to as it relates to the DISPLAY environment variable. For now, it is important to know that the display number will appear in the name of the log file associated with that display.

If the display doesn't start or acts strangely, then the log file for the display can be viewed: the /var/log/Xorg.0.log file for display 0, the /var/log/Xorg.1.log file for display 1, etc. Lines within these files are tagged with the following letters to indicate their importance:

  • II = Informational
  • WW = Warning
  • EE = Error
  • ** = From the configuration file
  • ++ = From the command line
  • -- = Probed
  • NI = Not implemented
  • ?? = Unknown

The lines that are of most interest for troubleshooting a non-working X server are those which indicate either a warning or an error. Before describing typical errors in starting the X server and their causes, it will be helpful to understand what the /etc/X11/xorg.conf file contains.

3.3.1 /etc/X11/xorg.conf File

The /etc/X11/xorg.conf file, along with any file with a *.conf extension in the /etc/X11/xorg.conf.d/ directory, contains numerous sections with configuration items. The most recent versions of the X server can actually run without any xorg.conf file as they automatically configure themselves when started, but an xorg.conf may still be used to create custom configurations. The following is a list of the sections that are defined for this file:

Section Purpose
Files

File pathnames for fonts and modules
ServerFlags

Server flags are global options
Module

Dynamic module loading of modules that extend the server
Extensions

Extension enabling for the X11 protocol
InputDevice

Input device description for keyboard and pointers
InputClass

Input class description
Device

Video card device description
VideoAdaptor

Xv video adaptor description
Monitor

Monitor description
Modes

Video modes descriptions
Screen

Screen configuration
ServerLayout

Overall layout combining other sections
DRI

Direct Rendering Infrastructure
Vendor

Vendor-specific configuration

A typical xorg.conf file will not contain all of these sections since many of them are optional. Also, some of these sections may be repeated multiple times to be able to support items such as multiple video cards, monitors, or input devices.

Note

The files in the conf.d directory are now the preferred method of configuring X. The xorg.conf file, if it exists, will be read first, then the files in conf.d. However, some distributions no longer provide the .conf file by default.

3.4 Troubleshooting the X Server

The http://www.x.org/wiki/FAQErrorMessages web page describes some common error messages seen when attempting to start the X server. The following examples describe some common errors and how to troubleshoot them based upon this information by modifying the /etc/X11/xorg.conf file.

3.4.1 Missing Configuration File

If a system does not currently have an /etc/X11/xorg.conf file or to create a new configuration file, generate it using the command:

root@localhost:~# X -configure

If the X server has not been stopped, then an error message similar to the following will appear:

Fatal server error:
Server is already active for display 0
If this server is no longer running, remove /tmp/.X0-lock
⁠⁠ 
and start again.

If the X server was not running when the command was executed, then a file named xorg.conf.new should be generated in the current directory. To test if this newly-generated file will work, first make a backup copy of the /etc/X11/xorg.conf file, then move the xorg.conf.new file to the /etc/X11/xorg.conf file and start the X server:

root@localhost:~# cp /etc/X11/xorg.conf /etc/X11/xorg.conf.backup
root@localhost:~# mv xorg.conf.new /etc/X11/xorg.conf
root@localhost:~# startx

The backup copy may be useful for troubleshooting X problems. In some cases, it may be useful to reset the X configuration to a “previous state”.

3.4.2 No Video Card Detected

The most serious type of error is No devices detected. This error indicates that the video card device specified in the Device section of the file may not be compatible with the video card in your system. A computer is more likely to have problems if it has very old, very new, or unusual hardware.

If your video card is made by Nvidia or AMD (formerly ATI), then it may not work with the Open Source driver from your Linux distribution. In this case, it may be possible to obtain a proprietary driver from the vendor’s website; however, there is no guarantee this driver will work either.

If your video card will not work with a proprietary driver, then it may be possible to use a generic driver known as vesa, which can drive most video cards that are compatible with VESA (Video Electronics Standards Association) standards. Using the vesa driver may not allow you to take full advantage of all the features of your video card; however, it should provide basic video functionality. The output below demonstrates the Driver section of an /etc/X11/xorg.conf file that has been configured to use the generic vesa driver:

Section "Device"
    Identifier  "Card0"
    Driver      "vesa"
EndSection

3.4.3 No Display Monitor Detected

The video display monitor is almost as crucial to a functioning X window system as the video card. If the screen stays dark and the X Windows cursor never appears, the problem is likely the video card, the display monitor, or both. Both the video card and monitor have to work together to display images. In order for them to work together, they must be able to synchronize on different display modes. When the video card and display monitor are unable to sync together, the error message from the log file will be something like the following:

(EE) Screen(s) found, but none have a usable configuration.

There are many factors that can limit the compatibility between a video card and a monitor:

  1. Horizontal sync rate
  2. Vertical refresh rate
  3. Color depth
  4. Resolution

As mentioned previously, a computer is more likely to have problems if it has very old, very new, or unusual hardware. It is especially likely to have problems when combining a very old video card with a very new monitor, or vice versa. For example, a very old video card may not have enough memory to even support the lowest resolution of a very new (and large) monitor. Likewise, a very new video card might drive the image signal at a faster rate, greater color depth or higher resolution than what a very old monitor can handle.

3.4.4 Fonts Unavailable

An error message like font named "fixed" cannot be found or loaded is fatal because if a required font cannot be loaded, then the X server is unable to start. This error message was more common years ago before fonts were built-in to the X server. It was previously necessary to ensure that the paths to the fonts on the system were properly configured in the Files section of the /etc/X11/xorg.conf file, which would look something like the following:

Section "Files"
    FontPath   "/usr/X11R6/lib/X11/fonts/misc/"
    FontPath   "/usr/X11R6/lib/X11/fonts/Speedo/"
    FontPath   "/usr/X11R6/lib/X11/fonts/Type1/"
    FontPath   "/usr/X11R6/lib/X11/fonts/CID/"
    FontPath   "/usr/X11R6/lib/X11/fonts/75dpi/"
    FontPath   "/usr/X11R6/lib/X11/fonts/100dpi/"
EndSection

Another problem that may have occurred in the past was that a separate X font service xfs was used to provide the fonts for the X server. If the xfs service was not started at the time the X server was started, then fonts would also be unavailable.

Although the use of the xfs service is now deprecated, you should be aware of this service for the LPIC exam. If the xfs is in use, then the Files section of the /etc/X11/xorg.conf file may appear with an entry that refers to a Unix socket with “unix:/7100” specified as the FontPath:

Section "Files"
   ModulePath   "/usr/lib/xorg/modules"
FontPath        "unix/:7100"          # local font server
# if local font server has problems, fall back on these
FontPath        "/usr/lib/X11/fonts/misc"
FontPath        "/usr/lib/X11/fonts/cyrillic"
FontPath        "/usr/lib/X11/fonts/75dpi/"
FontPath        "/usr/lib/X11/fonts/100dpi/"
FontPath        "/usr/X11R6/lib/X11/fonts/sgi"
FontPath        "/usr/lib/X11/fonts/Type1"
FontPath        "/usr/lib/X11/fonts/CID"
FontPath        "/usr/lib/X11/fonts/100dpi:unscaled"
FontPath        "/usr/lib/X11/fonts/75dpi:unscaled"
EndSection

Because the X server now has its own built-in fonts for the required fonts, the font error mentioned previously may not occur. Be familiar with what the Files section looks like for a current X server, which does not use an xfs service:

Section "Files"
ModulePath   "/usr/lib/xorg/modules"
FontPath     "catalogue:/etc/X11/fontpath.d"
⁠⁠ 
FontPath     "built-ins"
EndSection

Systems that do not use the xfs server use a newer way of finding fonts known as FontConfig. To install a new font for all users on a system that uses FontConfig, the font files should be copied to a subdirectory of the /usr/share/fonts directory. Subsequently, the administrator can execute the fc-cache command to update the font information cache of the system. For example, if new fonts are copied to the /usr/share/fonts/local directory, then it is possible to execute the command:

root@localhost:~# fc-cache /usr/share/fonts/local

For font files to be installed for an individual user, the fonts can be copied to a .fonts directory in the user's home directory. Then, the user can update their font information cache by executing the following command:

root@localhost:~# fc-cache ~/.fonts

3.4.5 ~/.xsession-errors

When applications are started via an icon in X Windows vs the command line, output going to standard error (STDERR) can no longer display on the screen. For some desktop environments (such as Gnome), the output from STDERR is sent to the ~/.xsession-errors log file, which can be used to troubleshoot problems encountered. The STDERR output from X Windows and the applications running through X Window are both sent to the ~/.session-errors file, so it is worth looking to see if ~/.xsession-errors exists in your home directory when troubleshooting any X Window problems.
