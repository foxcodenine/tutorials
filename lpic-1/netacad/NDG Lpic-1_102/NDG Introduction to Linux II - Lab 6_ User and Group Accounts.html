6.5 Step 5

The /etc/shadow file contains user account information related to passwords and password security policies. Note that regular users cannot view the contents of this file. To verify that the file is not viewable by a non-root account, try to execute the following command:

more /etc/shadow

Note the error message that is displayed:

sysadmin@localhost:~$ more /etc/shadow                                          
more: cannot open /etc/shadow: Permission denied

6.9 step 9

Create a user account named billy by executing the following command:

useradd billy
root@localhost:~# useradd billy

The useradd command also requires root privileges.

6.10 Step 10

Note that the billy account initially doesn’t have a password. Verify this by executing the following command:

grep billy /etc/shadow
root@localhost:~# grep billy /etc/shadow                                          
billy:!:18219:0:99999:7:::

The ! character in the second field indicated this account has no password. As a result, no user can log into this account.

6.11 Step 11

To assign a password, execute the following passwd command:

passwd billy

When prompted, give a new password of netlab123. You will be asked to enter this password twice:

root@localhost:~# passwd billy                                                    
Enter new UNIX password:                                                        
Retype new UNIX password:                                                       
passwd: password updated successfully

6.12 Step 12

To see the new password for this user, execute the following command:

grep billy /etc/shadow
root@localhost:~# grep billy /etc/shadow                                          
billy:$6$2BVAf4pk$FP3dgqmMZy6l9S1DzV.ssUypMXC2/PrWIu5ljdH0/iMOTFVg.WplhW43DhsK291Y5J/Qjf.74OmlQQ6H/Ujmf0:18219:0:99999:7:::

Note that the second field doesn’t contain netlab123, but rather an encrypted version of this password. The encrypted password displayed on your screen will be different than the encrypted password displayed in this screen shot because the encryption technique provides a different result each time you create a password, even if the same password is provided.

6.13 Step 13

To lock the billy account, execute the following command:

passwd –l billy
root@localhost:~# passwd -l billy                                                 
passwd: password expiry information changed.

To verify that the account is locked, execute the following command:

grep billy /etc/shadow
root@localhost:~# grep billy /etc/shadow                                          
billy:!$6$2BVAf4pk$FP3dgqmMZy6l9S1DzV.ssUypMXC2/PrWIu5ljdH0/iMOTFVg.WplhW43DhsK291Y5J/Qjf.74OmlQQ6H/Ujmf0:18219:0:99999:7:::

The exclamation ! point at the beginning of the password field indicates the account has been locked. The old password is still recoverable because removing this exclamation ! point would result in the password entry that was identical to the original encrypted password.

6.14 Step 14

To unlock the billy account, execute the following command:

passwd –u billy
root@localhost:~# passwd -u billy                                                 
passwd: password expiry information changed.

To verify that the billy account is unlocked, execute the following command:

grep billy /etc/shadow
root@localhost:~# grep billy /etc/shadow                                          
billy:$6$2BVAf4pk$FP3dgqmMZy6l9S1DzV.ssUypMXC2/PrWIu5ljdH0/iMOTFVg.WplhW43DhsK291Y5J/Qjf.74OmlQQ6H/Ujmf0:18219:0:99999:7:::

6.15 Step 15

To remove the billy user’s password completely, execute the following command:

passwd –d billy
root@localhost:~# passwd -d billy                                                 
passwd: password expiry information changed.

To verify this change took effect, execute the following command:

grep billy /etc/shadow
root@localhost:~# grep billy /etc/shadow                                          
billy::18219:0:99999:7:::
Accounts with no password pose a considerable security risk on the system. This sort of account should be created only when necessary, such as when a user requires a password reset, and should only exist for a very short period of time.

6.23 Step 23

To see which group the current user is a member of, execute the following command:

groups
root@localhost:~# groups                                                        
root

6.24 Step 24

To switch to a new primary group, execute the following newgrp command and then use the groups command to verify:

newgrp bin
groups
root@localhost:~# newgrp bin                                                    
root@localhost:~# groups                                                        
bin root

In the previous output, the first group bin indicates your primary group. Any new file that you create will be a member of this primary group. Typically, you need to be the member of a group to switch to it, but the root user can switch to any group by default.

6.25 Step 25

Execute the following commands to switch back to your original group and verify this change:

exit
groups
root@localhost:~# exit                                                          
exit                                                                            
root@localhost:~# groups                                                        
root

The newgrp command started a new shell and placed your account into a new group. The exit command closes the new shell, placing you back into the original shell. Be careful to not execute the exit command again at this time as that will exit the root shell and place you back into the sysadmin account.

6.26 Step 26

Files from the /etc/skel directory are automatically copied into a new user's home directories when the new account is created. To verify this, first execute the following command to place a file in the /etc/skel directory:

cp /usr/share/dict/words /etc/skel
root@localhost:~# cp /usr/share/dict/words /etc/skel

Next, execute the following command to create a new user account:

useradd -d /home/tim -m tim
root@localhost:~# useradd -d /home/tim -m tim

Finally, verify that the words file is now in the new user's home directory by executing the following command:

ls -l /home/tim
root@localhost:~# ls -l /home/tim                                               
total 952                                                                       
-rw-r--r-- 1 tim tim 971578 Nov 19 22:45 words

6.27 Step 27

Forcing users to change their passwords on a regular basis is part of a good security policy. To force the tim user to change his password every 60 days, use the chage command. Start by verifying that the fifth field of the /etc/shadow file has a value of 90, meaning the user isn't forced to change his password for 90 days:

grep tim /etc/shadow
root@localhost:~# grep tim /etc/shadow                                          
tim:!:18219:0:90:7:::
Recall that the value is currently 90 since it was changed previously in /etc/login.defs.

6.28 Step 28

Next, execute the following commands to change the maximum password field and verify the change:

chage -M 60 tim
grep tim /etc/shadow
root@localhost:~# chage -M 60 tim                                               
root@localhost:~# grep tim /etc/shadow                                          
tim:!:18219:0:60:7:::

6.29 Step 29

In 60 days, if the tim user doesn't change his password, his account will be locked out. To have the system issue a warning at login, use the -W option to the chage command. To permit a grace period in days, use the -I option to the chage command. Execute the following commands:

grep tim /etc/shadow
chage -W 10 -I 15 tim
grep tim /etc/shadow
root@localhost:~# grep tim /etc/shadow                                          
tim:!:18219:0:60:7:::                                                            
root@localhost:~# chage -W 10 -I 15 tim                                         
root@localhost:~# grep tim /etc/shadow                                          
tim:!:18219:0:60:10:15::

The warning field of 10 means issue a warning to the user that the password for the account will expire if not changed within the next 10 days. If the user ignores the warning, the account will be locked. The grace period field of 15 allows the user to log in after the expiration date, provided no more than 15 days have passed since the account was locked.

6.30 Step 30

To set an expiration date for the tim user account and verify the new value, execute the following commands:

chage -E "2025-12-12" tim
grep tim /etc/shadow
root@localhost:~# chage -E "2025-12-12" tim                                     
root@localhost:~# grep tim /etc/shadow                                          
tim:!:18219:0:60:10:15:20434:

6.31 Step 31

Recall that the expiration date is represented by the number of days since January 1, 1970 and the date the account is to expire. This value is hard to understand by viewing the /etc/shadow file directly, so execute the following command to view this value (and others) in a more human-readable format:

chage -l tim
root@localhost:~# chage -l tim                                                  
Last password change                                    : Nov 19, 2019          
Password expires                                        : Jan 18, 2020          
Password inactive                                       : Feb 02, 2020          
Account expires                                         : Dec 12, 2025          
Minimum number of days between password change          : 0                     
Maximum number of days between password change          : 60                    
Number of days of warning before password expires       : 10

6.32 Step 32

To modify a user, use the usermod command. For example, execute the following command to add the tim user to another group:

usermod -G sysadmin tim
root@localhost:~# usermod -G sysadmin tim

Verify the tim user's new group membership by executing the following command:

groups tim
root@localhost:~# groups tim                                                    
tim : tim sysadmin

6.33 Step 33

Change the tim user's login shell and verify the change by executing the following commands:

grep tim /etc/passwd
usermod -s /bin/tcsh tim
grep tim /etc/passwd
root@localhost:~# grep tim /etc/passwd                                          
tim:x:1007:1010::/home/tim:/bin/bash                                                        
root@localhost:~# usermod -s /bin/tcsh tim
root@localhost:~# grep tim /etc/passwd 
tim:x:1007:1010::/home/tim:/bin/tcsh

6.34 Step 34

To delete the tim user account, along with the user's home directory and mail file, and verify that the account has been deleted, execute the following commands:

userdel -r tim
grep tim /etc/passwd
root@localhost:~# userdel -r tim                                                                              
root@localhost:~# grep tim /etc/passwd

The lack of output from the grep command indicates that the user account no longer exists. Recall that the –r option also deletes the user’s home directory and mail file. Without the –r option, the account would have been deleted, but these files and directories would not have been removed.

6.35 Step 35

Create a new group and verify by executing the following commands:

groupadd mygroup
grep mygroup /etc/group
root@localhost:~# groupadd mygroup                                              
root@localhost:~# grep mygroup /etc/group     
mygroup:x:1010:

6.36 Step 36

Change the name of this group and verify the change by executing the following commands:

It may be worthwhile to mention that the group GID does not change when the group is renamed. The group number when running through the lab is 1010 since there have been 2 accounts added and remaining to this point.
groupmod -n ourgroup mygroup
grep ourgroup /etc/group
root@localhost:~# groupmod -n ourgroup mygroup                                  
root@localhost:~# grep ourgroup /etc/group                                      
ourgroup:x:1010:

6.37 Step 37

Delete the group and verify that it has been deleted by executing the following commands:

groupdel ourgroup
grep ourgroup /etc/group
root@localhost:~# groupdel ourgroup                                             
root@localhost:~# grep ourgroup /etc/group

The lack of output for the grep command indicates that the ourgroup group no longer exists.
