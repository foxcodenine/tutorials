9.17 Step 17

You will be modifying the /etc/ntp.conf file to enable the NTP service to use the local system as a time server by adding a server entry line for the localhost address. You will also add a line to specify when the system will switch to orphan mode. The tos orphan 6 line indicates that the system will switch to orphan mode if no strata 5 or lower servers can be reached.

First, execute the following command to edit the /etc/ntp.conf file using the vi editor:

vi /etc/ntp.conf
[root@centos ~]# vi /etc/ntp.conf

# For more information about this file, see the man pages
# ntp.conf(5), ntp_acc(5), ntp_auth(5), ntp_clock(5), ntp_misc(5), ntp_mon(5).

driftfile /var/lib/ntp/drift

#Permit time synchronization with our time source, but do not
# permit the source to query or modify the service on this system.
Restrict default nomodify notrap nopeer noquery

#Permit all access over the loopback interface. This could
# be tightened al well, but to do so would effect some of 
# the administrative functions.
restrict 127.0.0.1 
restrict ::1

# Hosts on local network are less restricted.
#restrict 192.168.1.0 mask 255.255.255.0 nomodify notrap

# Use public servers from the pool.ntp.org project.
# Please consider joining the pool (http://www.pool.ntp.org/join.html).
server 0.centos.pool.ntp.org iburst
server 1.centos.pool.ntp.org iburst
server 2.centos.pool.ntp.org iburst
server 3.centos.pool.ntp.org iburst
“/etc/ntp.conf” 58L, 200C

After pressing the Enter key to launch the vi editor, use the G command to go to the end of the file and the o command to start insert mode on the line below. Then, add the following lines to the file:

server 127.0.0.1
tos orphan 6
...
# Disable the monitoring facility to prevent amplification attacks using ntpdc
# monlist command when default..

disable monitor
server 127.0.0.1
tos orphan
-- INSERT --

Save and exit the /etc/ntp.conf file:

Esc
:wq

Finally, use the following command to verify the changes to the /etc/ntp.conf file:

tail -n 2 /etc/ntp.conf
[root@centos ~]# tail -n 2 /etc/ntp.conf
server 127.0.0.1
tos orphan

If your output does not match the output above, an empty newline may have been added while making edits to the /etc/ntp.conf file. In this case, simply modify the number of lines displayed by the tail command. For example, tail -n 3. If this is not the case, and you are still not seeing the output, go back to the previous steps and review your edits to the /etc/ntp.conf file.

9.21 Step 21

Chrony is a set of programs, the chrony daemon (chronyd) and the chrony command line tool (chronyc), that are used to ensure that the clock on a system is accurate. Remember that a daemon is just a process not owned by any user. A daemon is sometimes like a teenager; it only has a few states of existence: most of the time it sleeps, it wakes, it may do some work (like chores), and sometimes before it can work, it has to wait for a sibling. The chronyd daemon is a process that runs in the background. The chronyc tool is its command-line interface. Chronyd can get its reference time from a number of sources: from NTP servers (which are covered in the chapter), a human (from the keyboard using chronyc commands), or from the hardware clock (RTC) at boot time. For servers that only have intermittent access to the internet, like those that only have modem access - yes, there are still places in the world without broadband - chronyd is perfect for synchronizing time.

Like many daemons, chronyd is configured through a text-based file. Many of these text-based configuration files are kept in the /etc directory. The configuration file for chronyd is /etc/chrony.conf.

more /etc/chrony.conf
[root@centos ~]# more /etc/chrony.conf

The chronyd daemon can get its time from the same NTP servers discussed in the chapter, so the first thing you will see in the chrony.conf file is a list of the NTP servers:

...
server 0.centos.pool.ntp.org iburst
server 1.centos.pool.ntp.org iburst
server 2.centos.pool.ntp.org iburst
server 3.centos.pool.ntp.org iburst
...

To synchronize to the NTP server, the iburst utility can be used to send a burst of eight packets when the server is unreachable. It will then shorten the time until the next iburst, until it synchronizes. It’s like it becomes impatient and sends the next burst at a shorter amount of time. It will keep shortening the time between bursts until it synchronizes with the NTP server. Some public NTP servers consider the use of iburst to be aggressive.

Some of the important fields in the output above are:

  • Reference ID: This is the reference ID and name (or IP address) if available, of the server to which the computer is currently synchronized.

  • Stratum: The stratum indicates how many hops you are away from a computer with an attached reference clock. In your organization you may set up stratum layers in a hierarchy to reduce the time sync traffic.

  • Ref time: This is the time (in UTC) at which the last measurement from the reference clock was used to make adjustments to the time.

9.25 Step 25

Next, use the sources command to view information about the chrony synchronization:

chronyc sources
[root@centos ~]# chronyc sources
210 Number of sources = 4
MS Nem/IP address          Stratum Poll Reach LastRx Last sample
============================================================================
^+ web.vranetworks.net           3   8  377    124   -962us[ -963us] +/-  58ms
^- blue.1e400.net                3   7  377    121  +4412us[+4411us] +/-  79ms
^+ 198.255.68.106                2   8  377    251  +1409us[+1407us] +/-  72ms
^* callisto.netbunker.org        2   8  377    120   -375us[ -375us] +/-  27ms

The output of the command above shows the servers that chrony can synchronize to. The columns in the output are:

  • M:

    MS Nem/IP address          Stratum Poll Reach LastRx Last sample
============================================================================

    This column shows the mode of the source. ^ means a server, = means a peer and # indicates a locally-connected reference clock.

  • S:

    MS Nem/IP address          Stratum Poll Reach LastRx Last sample
============================================================================

    This shows the state of the sources. An asterisk * character identifies where chronyd is currently synchronized. A plus + sign signifies other acceptable sources. A question ? mark signifies a source where connection has been lost. An x after the carat, means chronyd thinks that source is a falseticker: a source whose time is inconsistent with other sources. If there is a tilde ~ character after the carat, that means chronyd believes that source has too much variability in its time. The ~ condition may also be shown at start-up until at least three time samples have been gathered from it and it appears stable.

  • Nem/IP address:

    MS Nem/IP address          Stratum Poll Reach LastRx Last sample
============================================================================

    This is the host name and IP address.

  • Stratum:

    MS Nem/IP address          Stratum Poll Reach LastRx Last sample
============================================================================

    These are the hierarchy of time reference layers in your organization. Stratum 1 shows a computer locally connected to a time reference clock. If a computer is synchronized to a stratum 1 computer, then it’s in stratum 2. A computer synchronized to a stratum 2 computer is in stratum 3, and so on.

  • Poll:

    MS Nem/IP address          Stratum Poll Reach LastRx Last sample
============================================================================

    This shows the rate at which the source is being polled, as a base-2 logarithm of the interval, in seconds. For example, if you see a number 6, that means it was last polled at 2^6 = 64 seconds ago. Chronyd will vary the polling time depending on network bandwidth conditions.

  • Reach:

    MS Nem/IP address          Stratum Poll Reach LastRx Last sample
============================================================================

    This shows how easily the source is reached, set as an octal number. It keeps a register of 8 bits and is updated every time packets are received or missed from the source. A value of 377 indicates that a valid reply was received from all of the last eight transmissions.

  • LastRX:

    MS Nem/IP address          Stratum Poll Reach LastRx Last sample
============================================================================

    This shows how long it has been since the last sample was received from a source. The value given if often in seconds. If you see the letters m, h, d or y, that means minutes, hours, days, or years, respectively. A value of 10 years means that no sample has ever been received from that source.

  • Last sample:

    MS Nem/IP address          Stratum Poll Reach LastRx Last sample
============================================================================

    This value is the offset or difference between the local clock and the last time reference received from a source. The value in square [] brackets is the actual offset given in ns=nanoseconds, us=microseconds, ms=milliseconds, or s=seconds. The number to the left of the square brackets shows the original time measurement, adjusted to allow for any slews applied to the local clock since. The number just after the +/- indicator shows the margin of error in the measurement. Positive offsets show that the local clock is a little faster than the source.

9.26 Step 26

The hardware clock is known as the Real Time Clock (RTC) or CMOS clock. There is a battery on the motherboard that keeps the hardware clock going when the computer is turned off. System time comes from the system clock. The system clock is set from the hardware clock when the computer boots and can be adjusted from the /etc/adjtime file.

View the contents of the /etc/adjtime file by executing the following command:

9.27 Step 27

The timedatectl command is used to view and control time on systems using systemd as their init-system.

You can directly change the system time with the timedatectl command using the following syntax:

timedatectl set-time “yyyy-MM-dd hh:mm:ss”

For example:

timedatectl set-time “2025-11-09 11:13:54”

The example command above will result in an error message since automatic time synchronization is enabled in our virtual environment.

The hardware clock time is either set to local time or to Coordinated Universal Time (UTC). It is best practice to the hardware clock time to UTC so you don’t have to deal with time zones.

 RTC in local TZ: yes
Warning: The system is configured to read the RTC time in the local time zone. 
         `timedatectl set-local-rtc 0’.

Note in the output of the above command that the hardware clock is not in UTC but in local time. This is not good practice.

9.29 Step 29

To change the hardware clock to UTC, use the following command:

timedatectl set-local-rtc 0
timedatectl | grep local
[root@centos ~]# timedatectl set-local-rtc 0
[root@centos ~]# timedatectl | grep local
 RTC in local TZ: no
